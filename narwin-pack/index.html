narwin-pack

And why we built a PostCSS package

Cory Tanner / @ctannerweb / @dockyard

Why PostCSS?

  • Main reason is the ability to pick and choose plugins you want to use
  • You can set PostCSS up to act like Sass
  • Is modular so the preprocessor is 3-30 times faster
  • Create custom Plugin Packs that you can easily use for work like we did with narwin-pack

PostCSS Plugins

  • Take normal css and modifies it based on the rules in the plugin
  • You can write plugins yourself
  • Dont expect one plugin to solve every need
  • The beauty with plugins is that there will always be new plugins that solve new problems

PostCSS Packs

  • A collection of plugins that solve a bigger issue
  • cssnext allows you to use tomorrow's CSS syntax now
  • narwin-pack are hand picked plugins that meet the needs of Dockyards UX development process

narwin-pack Plugins

  • autoprefixer
  • postcss-partial-import
  • postcss-nested
  • postcss-custom-properties
  • postcss-calc

autoprefixer

  • Based on caniuse rules and the plugin will auto add the needed prefix of a style
  • Very helpfull with transforms and transitions

postcss-partial-import

  • Lets you set up modular .css files and import them all into one app.css file
  • Who doesn't like modular CSS!

postcss-nested

  • Unwraps nested rules how Sass would
  • We only use this for pseudo classes and media queries
  • You loose search options if you nest BEM classes like .block { &__element { display: flex; } }

postcss-custom-properties

  • Variables in your CSS!
  • We have a load.css file that has all our vars like :root { --green: rgb(61, 154, 104); } color: var(--green);

postcss-calc

  • Reduces calc() references when possible
  • Any calculations with variables are now possible :root { --base-font-size: 16px; } font-size: calc(var(--main-font-size) * 2);
